# Sprintpilot-BMAD-Module — Concept & Plan

> Status: **Concept / proposal**. Not scheduled. Saved 2026-05-22 for later execution.
>
> Goal: repackage Sprintpilot from a "sibling addon" npm package into a proper **BMAD module** installable via `npx bmad-method install` — the same shape as the TEA module (`bmad-method-test-architecture-enterprise`).

---

## 1. Context & motivation

Today Sprintpilot ships as `@ikunin/sprintpilot`, a standalone npm package that users install **alongside** an existing BMAD installation. The package's own CLI (`lib/commands/install.js`) does:

- per-tool skill deployment (`.claude/skills/`, `.cursor/skills/`, …)
- `AGENTS.md` system-prompt block management
- gitignore upserts
- copying its runtime into `_Sprintpilot/`

This works but it's a parallel installer. BMAD has its own installer (`npx bmad-method install`) with a module picker, a registry, and a coherent contract for external modules. **TEA proves third-party modules are first-class.** Sprintpilot can follow the same pattern.

Benefits of being a real module:
- Single install path users already know
- Discoverable in the BMAD module picker
- Versioning and update flows align with BMAD's
- Optional dual-publish as a Claude Code plugin (TEA does this) → second user-acquisition channel

---

## 2. Verdict

**Strongly viable.** TEA is the proof-of-concept. Sprintpilot is structurally similar (skills + workflows + a runtime), so the work is mostly relocation + adapter authoring, not re-architecture.

**Estimated effort**: ~2 weeks technical work + ~1 week upstream PR cycle.

## 3. BMAD module contract (verified from BMAD-METHOD source)

Source: `github.com/bmad-code-org/BMAD-METHOD` and `github.com/bmad-code-org/bmad-method-test-architecture-enterprise` (TEA).

### 3a. Module registry

`BMAD-METHOD/bmad-modules.yaml` (repo root) is the single source of truth for the installer picker. Current entries (TEA, BMad Builder, Automator, CIS, Game Dev Studio, WDS Expansion) all follow this schema:

```yaml
sprintpilot:                                       # registry key
  url: https://github.com/ikunin/sprintpilot
  module-definition: src/module.yaml               # path inside the repo
  code: sprintpilot                                # short code used in _bmad/<code>/
  name: "Sprintpilot — Autopilot Addon"
  description: "Autonomous sprint execution, git workflow automation, multi-agent intelligence"
  defaultSelected: false
  type: bmad-org                                   # or "community" once that exists
  npmPackage: "@ikunin/sprintpilot"
  default_channel: stable                          # stable | next
```

**Getting listed requires an upstream PR to BMAD-METHOD.** No self-registration mechanism observed (a `tools/migrate-custom-module-paths.js` exists, hinting at custom/local module support — worth investigating if upstream cooperation stalls).

### 3b. Installed payload shape (under `_bmad/<code>/`)

What BMAD's installer writes to the consumer project after install:

```
_bmad/<code>/
├── config.yaml           # generated by installer (resolved values from module.yaml prompts)
├── module-help.csv       # skill catalog (copied from src/)
├── agents/               # (optional) copied from src/
└── workflows/            # (or any other content dirs) copied from src/
```

`module.yaml` itself is **not copied** — it's consumed during install only.

### 3c. Per-skill files

Inside each skill directory:

- `SKILL.md` — frontmatter + body (unchanged from current Sprintpilot format)
- `bmad-skill-manifest.yaml` — type metadata
  - For agents (rich): `type: agent, name, displayName, title, icon, capabilities, role, identity, communicationStyle, principles, module, canonicalId, webskip, hasSidecar`
  - For workflow skills (minimal): `type: skill`
- `workflow.md` (workflows only) — tri-modal routing (Create / Edit / Validate)
- `workflow.yaml` (workflows only) — machine-readable workflow metadata
- `steps-c/`, `steps-e/`, `steps-v/` (workflows only) — step files per mode
- `templates/`, `resources/` (optional)

### 3d. Central registration (auto-managed)

After install, the BMAD installer maintains these in `_bmad/_config/`:

- `manifest.yaml` — modules list with `source: external`, `npmPackage`, version
- `skill-manifest.csv` — every skill registered with `install_to_bmad: "true"`
- `agent-manifest.csv` — every agent persona
- `files-manifest.csv` — every file with SHA-256 hash (drift detection)
- `bmad-help.csv` — flattened help index
- `ides/<ide>.yaml` — per-IDE configuration

The installer also generates `.claude/commands/<command>.md` launchers (and equivalents for Codex `.agents/skills/`, Cursor, Windsurf, etc.) — these are the per-tool slash-command bridges.

### 3e. `module.yaml` schema (verified from TEA)

```yaml
code: <short-code>
name: "<display name>"
header: "<one-line>"
subheader: "<sub-description>"
description: "<full description>"
default_selected: false

agents:                                  # optional
  - code: <persona-code>
    name: <persona-name>
    title: <persona-title>
    icon: "🎯"
    description: "<persona-pitch>"
    team: <team-key>                     # e.g. software-development

# Variable prompts — installer asks the user, writes resolved values to config.yaml
<var_name>:
  prompt: "<question>"
  default: <default-value>               # string | boolean | number
  result: "{value}"                      # or "{project-root}/{value}", etc.
  single-select:                         # optional
    - value: "<choice>"
      label: "<display label>"
    - value: "<choice>"
      label: "<display label>"

# Conditional post-install messaging
post-install-notes:
  <var_name>:
    <chosen-value>: |
      <multi-line setup instructions shown after install>

# Directories to create at install time (declarative)
directories:
  - "{var_name}"
```

Available template variables: `{value}`, `{project-root}`, `{output_folder}`, `{user_name}`, plus any previously-defined variable in the same module.yaml.

## 4. TEA reference pattern (the precedent)

TEA's repo layout (`github.com/bmad-code-org/bmad-method-test-architecture-enterprise`):

```
bmad-method-test-architecture-enterprise/
├── .claude-plugin/marketplace.json     # Dual-publish as Claude Code plugin
├── src/
│   ├── module.yaml                     # ← THE config questionnaire
│   ├── module-help.csv
│   ├── agents/
│   │   └── bmad-tea/                   # Murat persona
│   └── workflows/
│       └── testarch/
│           ├── bmad-testarch-atdd/
│           ├── bmad-testarch-ci/
│           └── … (9 workflows total)
├── tools/                              # Dev tools (validators, doc builders) — NOT runtime
├── test/                               # Schema and install tests
├── package.json                        # NO bin, NO main, NO postinstall
├── README.md
└── LICENSE                             # MIT
```

Key observations:
- **`package.json` has no `bin`, no `main`, no `postinstall`** — it's a pure content package
- The BMAD installer does all the work; the package just provides files
- Skills are wholly self-contained (no cross-package imports at runtime)
- `.claude-plugin/marketplace.json` declares the same skills for direct Claude Code plugin install — enables Path 2 below

## 6. Per-skill changes required

For each of the 11 skills:

1. **Add `bmad-skill-manifest.yaml`** (minimal):
   ```yaml
   type: skill
   ```

2. **Update path references**: search/replace `_Sprintpilot/` → `_bmad/sprintpilot/` in:
   - All `SKILL.md` bodies
   - All `workflow.md` / `workflow.orchestrator.md` files
   - All scripts that read other Sprintpilot files

3. **Optional `workflow.yaml`** (TEA-style machine-readable metadata) for skills that warrant tri-modal routing. Most Sprintpilot skills don't need Edit / Validate modes — only Create.

4. **No SKILL.md body rewrites needed** — the prompt content is portable.

### Path 1 — BMAD installer (full integration)

```bash
npx bmad-method install
# Multi-select picker shows:
#   [x] BMad Method (core, bmm)            ← built-in
#   [ ] BMad Test Architect (tea)
#   [x] Sprintpilot — Autopilot Addon      ← new entry from bmad-modules.yaml

# Copies skill content to .claude/skills/<skill>/ (and equivalents per IDE)

```

```bash

# No _bmad/ integration; lighter usage (skills work but autopilot state files live wherever
# the user configures them)
```

Both paths from the same npm package. Two acquisition channels.

## 9. Backwards compatibility

Existing v2 users have `_Sprintpilot/` at project root. The v3.0.0 BMAD-module migration is a breaking shape change.

Migration paths:

### Option A — Manual migration (documented in CHANGELOG)

User runs `npx @ikunin/sprintpilot migrate-to-v3` which:
1. Detects `_Sprintpilot/` at project root
2. Snapshots user-customized configs (`modules/*/config.yaml`, profile overrides)
3. Removes `_Sprintpilot/`
4. Prompts user to run `npx bmad-method install` and select Sprintpilot
5. Re-applies snapshotted customizations to the new `_bmad/sprintpilot/config.yaml`

### Option B — Automatic migration on next `bmad install`

If BMAD's installer can detect the legacy `_Sprintpilot/` directory and auto-handle the migration. Requires upstream cooperation or a Sprintpilot-side preflight hook.

**Recommendation**: Option A. Mirrors the existing v1 → v2 migration logic in `lib/commands/install.js`.

## 12. Strategic considerations

### Why this is worth doing

1. **User journey simplification** — `npx bmad-method install` becomes the single installer. Today: two installers (BMAD's + Sprintpilot's).
2. **Discoverability** — appearing in the BMAD picker reaches users who'd never search for "@ikunin/sprintpilot".
3. **Ecosystem alignment** — proves Sprintpilot follows BMAD's conventions rather than parallels them.
4. **Dual-publish to CC plugin marketplace** — second user-acquisition channel for ~30 minutes of extra work.
5. **Future-proofing** — when BMAD adds new install-time features (IDE additions, plugin marketplace tie-ins), Sprintpilot inherits them for free.

### Why we might NOT do it

1. **Loss of installer control** — today Sprintpilot's installer does fine-grained things (legacy v1 snapshot/restore, retention-aware backups, secrets-scan during commit) that BMAD's installer may not match. Some of this may need to move into Sprintpilot's runtime or a postinstall step.
2. **Upstream coupling** — Sprintpilot's release cadence becomes more dependent on BMAD's. Breaking changes in `module.yaml` schema or installer behavior would force Sprintpilot reworks.
3. **Naming overlap** — users currently search for "Sprintpilot". Under BMAD they'd search for it via BMAD's picker. Brand visibility drops slightly.

**Mitigation**: keep the standalone `@ikunin/sprintpilot install` path working as a Path 3 for at least one major version after the BMAD-module v3.0.0, with a deprecation notice. Gradual migration.
